LESSON 2.1 · SDLC MODELS
Testing in Different SDLC Models
The Software Development Lifecycle (SDLC) is the structured process for planning, creating, testing, and delivering software. Different SDLC models have different implications for how and when testing happens.

The key ISTQB principle: testing activities must be adapted to the SDLC model in use. There is no one-size-fits-all approach.
🌊 Sequential (Waterfall)
Testing happens at the end, after development is complete. High risk — defects found late are expensive. Each phase flows into the next with no overlap.
📐 V-Model
Each development phase has a corresponding test level. Testing planned in parallel with development. Defects found earlier than Waterfall.
🔄 Iterative / Incremental
Software built in cycles. Each iteration delivers working functionality. Testing happens within every iteration — not just at the end.
⚡ Agile (Scrum/Kanban)
Testing integrated into every sprint. QA works alongside developers. Continuous feedback, fast bug detection. TDD and ATDD common practices.
Q — QA DETECTIVE
"In Waterfall, testing is the last room in the house. By the time you get there, the foundation is already cracked. In Agile, every room gets inspected as it's built. Q prefers Agile. Fewer surprises."

LESSON 2.3 · TEST LEVELS
The 4 Test Levels — From Unit to Acceptance
ISTQB defines four test levels, each with a different scope, objective, and team responsible. Understanding them is critical for interviews.
🔬 Component Testing (Unit Testing)
Who: Developers · What: Individual functions/modules · Goal: Find defects in isolation. Fastest and cheapest level. Tools: JUnit, pytest, Jest.
🔗 Integration Testing
Who: Developers or QA · What: Interfaces between components · Goal: Find defects in interactions. API testing lives here.
🖥️ System Testing
Who: QA team · What: Complete integrated system · Goal: Verify the system meets requirements. Most functional testing happens here.
✅ Acceptance Testing (UAT)
Who: Users / Business · What: Business requirements · Goal: Confirm the system is ready for production. Final gate before go-live.
Q — QA DETECTIVE
"A defect found in component testing costs $1 to fix. The same defect found in production costs $100. This is why Q is always involved from day one. The earlier the detective, the cheaper the case."

LESSON 2.4 · TEST TYPES
Test Types — Functional vs Non-Functional
Test types classify testing by what they verify. They can be applied at any test level. ISTQB recognizes four main types:
⚙️ Functional Testing
Tests WHAT the system does. Verifies functions, features, and business rules. Example: "Does the login button work?" Black-box by nature.
📊 Non-Functional Testing
Tests HOW WELL the system performs. Performance, security, usability, reliability, compatibility. "How fast does it load under 1000 users?"
🏗️ Structural Testing
White-box testing. Tests the internal structure — code paths, branches, loops. Used to assess test coverage. Developers typically perform this.
🔄 Change-Related Testing
Testing after changes. Includes regression testing (nothing broke) and confirmation testing (bug is fixed).
Q — QA DETECTIVE
"Rookie mistake: thinking regression testing and retesting are the same. Retesting (confirmation testing) verifies a specific bug is fixed. Regression testing ensures fixing that bug didn't break anything else. Two different investigations."

LESSON 2.5 · MAINTENANCE TESTING
Maintenance Testing — Testing After Release
Maintenance testing occurs after a system is deployed to production. Changes to a live system — whether fixes, improvements, or migrations — require testing to ensure stability.
🔧 Corrective maintenance
Fixing defects reported in production. Requires confirmation testing + regression testing.
📈 Adaptive maintenance
Changing the system to work in a new environment (new OS, new database, new hardware). Impact analysis is critical.
✨ Perfective maintenance
Adding new features or improving existing ones after release. Full regression suite required.
🌐 Impact Analysis
Before any maintenance, assess what areas could be affected by the change. Determines regression scope. A bad impact analysis = missed bugs in production.

LESSON 2.6 · AGILE TESTING
Testing in Agile — The QA Role in Scrum
In Agile, testing is continuous and collaborative. QA is not a phase at the end — it's embedded in every sprint. The QA detective is in every ceremony, every standup, every planning session.
📋 Sprint Planning
QA reviews user stories, asks "how will we test this?", identifies acceptance criteria and edge cases before development starts.
🏃 During Sprint
QA tests stories as they're completed — not waiting for the end. Continuous feedback. TDD (Test-Driven Development) and BDD (Behaviour-Driven Development) practiced.
✅ Definition of Done
QA helps define DoD. A story is not "done" until it passes QA. DoD ensures consistent quality across all stories in every sprint.
🔄 Retrospective
QA contributes to process improvement. "What escaped QA this sprint? Why? How do we prevent it?" Continuous improvement of the testing process itself.
Q — QA DETECTIVE
"In Agile, QA who waits for development to finish is already too late. The best QA detective reads the requirements on day one and asks the uncomfortable questions — before a single line of code is written. That is shift-left in practice."
