LESSON 6.1 · TEST TOOL CATEGORIES
The Test Tool Landscape
ISTQB categorizes test tools by purpose. Knowing the categories — not just specific tools — is what the exam tests.
📋 Test Management
Organize test cases, track execution, link to requirements. Tools: Jira + Zephyr, TestRail, qTest, Azure DevOps Test Plans.
⚡ Test Execution / Automation
Execute tests automatically. UI: Selenium, Cypress, Playwright. API: Postman, RestAssured. Mobile: Appium, Espresso.
📊 Performance Testing
Simulate load and measure performance. Tools: JMeter, k6, Gatling, Locust. Used for load, stress, and soak testing.
🔒 Security Testing
Find vulnerabilities. SAST (code): SonarQube, Checkmarx. DAST (runtime): OWASP ZAP, Burp Suite. Dependency scanning: Snyk.
🐛 Defect Tracking
Log, track, and manage defects. Tools: Jira, GitHub Issues, Azure DevOps Boards, Bugzilla, MantisBT.
🔬 Static Analysis
Examine code without execution. Tools: SonarQube, ESLint, Pylint, PMD, Checkstyle. Integrated in CI/CD pipelines.

LESSON 6.2 · TEST AUTOMATION
Test Automation — Benefits, Risks, and Reality
Test automation uses tools to execute tests and compare actual results to expected results automatically. It is most valuable for regression testing and CI/CD pipelines.
✅ When to automate
Regression tests run frequently · Stable features unlikely to change · Repetitive data-driven tests · Smoke tests run on every build · API contract tests
❌ When NOT to automate
Exploratory testing · Usability testing · One-off tests · Frequently changing UI · Tests requiring human judgment · Accessibility assessment
⚠️ Automation risks
High initial investment · Maintenance cost when UI/API changes · False sense of security (passing tests ≠ no bugs) · Requires coding skills · Test debt accumulates
📈 ROI of automation
ROI increases over time as tests are reused. Break-even point: typically after 3–5 regression cycles. High-frequency, stable tests have the best ROI.
Q — QA DETECTIVE
"A 100% automated test suite that all passes is not a guarantee of quality. It is a guarantee that the code behaves exactly as it did when the tests were written. If the requirements were wrong, the tests are wrong too. Automation confirms consistency — not correctness."

LESSON 6.3 · TOOL SELECTION
Selecting and Introducing Test Tools
Choosing the wrong tool is a common and expensive mistake. ISTQB provides a structured approach to tool evaluation and adoption.
1
Assess organizational needs
What problem are you solving? What is your technology stack? What skills does the team have?
2
Evaluate against criteria
Compatibility with stack · Ease of use · Vendor support · Licensing cost · Community and documentation
3
Pilot on a real project
Test the tool on a real (but non-critical) project before full rollout. Measure ROI and identify integration challenges.
4
Roll out with training and support
Train the team. Define standards for tool use. Monitor adoption. Measure impact on quality and efficiency.

LESSON 6.4 · CI/CD AND TESTING
Testing in CI/CD Pipelines
In modern development, tests are automatically triggered by code changes. CI/CD (Continuous Integration/Continuous Delivery) pipelines run tests at every stage of the build and deployment process.
🔄 CI — Continuous Integration
Every code commit triggers: build → static analysis → unit tests → integration tests. Defects are detected within minutes. Developers fix immediately while context is fresh.
🚀 CD — Continuous Delivery
Automated pipeline deploys to staging/production after CI passes. Triggers: system tests, performance tests, security scans. One-click or automated deployment to production.
🧪 Test Pyramid
Many unit tests (fast, cheap) → fewer integration tests → fewer UI/E2E tests (slow, expensive). More tests at the bottom = faster feedback, lower maintenance cost.
⚡ Shift-Left in CI/CD
Static analysis and unit tests run on every commit. API tests on every build. E2E tests on every release candidate. Problems found in seconds — not days.
Q — QA DETECTIVE
"In a mature CI/CD pipeline, every code commit is a crime scene investigation. Static analysis is the first responder. Unit tests are the forensics team. Integration tests are the investigators. By the time it reaches production, it has been interrogated at every level."
