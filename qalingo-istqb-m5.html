LESSON 5.1 · TEST PLANNING
Test Planning — The Test Plan Document
A Test Plan is the master document for all testing activities. It defines what will be tested, how, by whom, when, and what success looks like. It is the detective's case brief before the investigation begins.
📋 What a Test Plan contains
Scope and objectives · Test approach and strategy · Test levels and types · Resources and roles · Schedule and milestones · Entry and exit criteria · Risks and mitigations
🎯 Entry and Exit Criteria
Entry: conditions to START testing (build ready, environment set up, test data available)
Exit: conditions to STOP testing (pass rate ≥ 95%, no critical bugs open)
⚡ In Agile
Test plans are lightweight and evolve. Often a "test strategy" section in each sprint. Planning happens in sprint planning. Exit criteria = Definition of Done.
📊 Test Estimation
Techniques: metrics-based (historical data), expert-based (experience), planning poker (Agile). Always include buffer for defect retesting and regression.

LESSON 5.2 · RISK-BASED TESTING
Risk-Based Testing — Test What Matters Most
Risk-based testing prioritizes testing activities based on risk. Since you can never test everything, you focus on areas where failure would be most likely and most damaging.

Risk = Likelihood of failure × Impact on business

High likelihood + High impact = test first, test thoroughly.
Low likelihood + Low impact = test less, or not at all.
🔴 Product Risk
Risk that the software will fail or not meet requirements. Examples: complex new feature, integration with external API, security-sensitive functionality, legacy code being modified.
⚙️ Project Risk
Risk that testing activities will be affected externally. Examples: environment not ready, test data unavailable, key team member sick, scope creep, tight deadline.
📊 Risk Assessment
Identify risks → Analyze likelihood and impact → Prioritize → Design tests for high-risk areas → Monitor risks throughout the project
📉 Risk Mitigation
Testing itself reduces product risk. The more thorough the testing of a high-risk area, the lower the residual risk of releasing it to production.

LESSON 5.3 · TEST MONITORING
Test Monitoring and Control
Test monitoring tracks test progress against the plan. Test control takes corrective actions when the project deviates from the plan.
📊 Key Metrics
Test cases planned vs executed · Pass rate % · Defect detection rate · Defect density · Test coverage % · Blocked tests (environment issues)
📈 Test Progress Reports
Regular reports to stakeholders. Show: what was tested, what passed/failed, open defects, risks, and whether the project is on track to meet exit criteria.
🔄 Control Actions
If behind schedule: reprioritize test cases, add resources, reduce scope. If defect rate is high: escalate, pause releases, add test cycles. Always communicate to stakeholders.
📋 Test Summary Report
Final report at end of testing. Summarizes what was tested, what was found, coverage achieved, risks remaining, and recommendation for release (go/no-go).

LESSON 5.4 · DEFECT MANAGEMENT
Defect Management — From Discovery to Closure
A defect lifecycle: New → Open → In Progress → Fixed → Retesting → Closed (or Rejected / Deferred)
📝 Defect Report contents
Unique ID · Title · Steps to reproduce · Expected result · Actual result · Environment · Severity · Priority · Status · Attachments (screenshots, logs)
🔴 Severity vs Priority
Severity = technical impact (set by QA)
Priority = business urgency (set by product/business)
Typo on homepage: Low severity, High priority
✅ Confirmation Testing
After a defect is fixed, re-run the original failing test to confirm it passes. This is NOT regression testing — it is verifying the specific fix.
🔄 Regression Testing
After fixing a defect, run the full (or risk-based) regression suite to ensure the fix did not break anything else. Different from — and in addition to — confirmation testing.

LESSON 5.5 · CONFIGURATION MANAGEMENT
Configuration Management in Testing
Configuration management ensures that test items, test environments, and test results are correctly identified, tracked, and controlled.

Without configuration management, you cannot reproduce test results, trace defects to specific builds, or know exactly what version of the system was tested.
🏷️ What to control
Software builds (versions) · Test environments · Test data · Test scripts · Test results · Defect reports
🔍 Why it matters for QA
Reproducibility: "Bug only on v2.3.1, not v2.3.2" · Traceability: link test cases to requirements · Auditability: prove what was tested and when
🛠 Tools
Version control: Git · Test management: Jira, TestRail, Zephyr · Environment management: Docker, Kubernetes · Build tracking: Jenkins, GitHub Actions
📋 Traceability Matrix
Links requirements → test cases → test results → defects. Shows exactly which requirements are covered, which tests cover them, and which defects were found.
