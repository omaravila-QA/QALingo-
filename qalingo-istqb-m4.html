LESSON 4.1 · BLACK-BOX TECHNIQUES
Black-Box Test Techniques
Black-box testing treats the system as a black box — you can only see inputs and outputs. You have no access to the internal code. Tests are derived from requirements, user stories, and specifications.

The four main ISTQB black-box techniques:
📦 Equivalence Partitioning (EP)
Divide inputs into partitions where all values should behave the same. Test ONE value per partition. If one fails, they all should fail — so one is enough.
📐 Boundary Value Analysis (BVA)
Test values AT boundaries and just OUTSIDE them. If a field accepts 1–100, test: 0, 1, 100, 101. Defects cluster at boundaries.
📊 Decision Table Testing
For systems with multiple conditions producing different actions. Each column = one rule (combination of conditions + action). Ensures all combinations are tested.
🔄 State Transition Testing
For systems with states (e.g. pending → active → suspended). Test valid transitions AND invalid ones. A common source of production bugs.
Q — QA DETECTIVE
"The exam trick: BVA always tests 4 values for a simple range — min-1, min, max, max+1. Always. If you see a question with a range and only 2 boundary values, the answer is wrong. Q has never seen an exception."

LESSON 4.2 · BOUNDARY VALUE ANALYSIS
BVA Deep Dive — The Most Tested Technique
BVA is the most frequently tested technique in the ISTQB exam. The rule is simple:

For a range min to max, test: min-1, min, max, max+1

For a field accepting ages 18–65: test 17, 18, 65, 66.
For a password of 6–12 characters: test 5, 6, 12, 13.

Why? Most boundary defects are "off by one" errors — the developer coded >= instead of > or used the wrong boundary value. BVA catches these systematically.
Q — QA DETECTIVE
"The 2-value vs 3-value BVA debate. ISTQB CTFL v4.0 uses 2-value BVA by default: min and max+1, min-1 and max. Some older material uses 3-value (min-1, min, min+1). In the exam, apply what the question implies. When in doubt, go with 2-value."

LESSON 4.3 · EXPERIENCE-BASED TECHNIQUES
Experience-Based Testing — Error Guessing and Exploratory
Experience-based techniques rely on the tester's knowledge, intuition, and past experience. They complement formal techniques and are especially valuable when time is limited or requirements are incomplete.
🎯 Error Guessing
Tester uses experience to predict likely defects. Often documented as a "defect attack list." Ask: "Where have bugs hidden before?" Empty fields, null values, special characters, race conditions.
🔍 Exploratory Testing
Simultaneous test design, execution, and learning. Guided by a charter. Adaptive — tester follows the evidence. Best used by experienced QAs on complex or poorly documented systems.
📋 Checklist-Based Testing
Tests derived from a checklist of items to verify. Covers high-level conditions. Provides consistency across testers. Less detailed than formal test cases.
⚡ When to use them
When time is short · When requirements are unclear · To supplement formal techniques · When testing legacy systems · During regression to find regressions formal tests miss

LESSON 4.4 · WHITE-BOX TECHNIQUES
White-Box Testing — Testing the Inside
White-box testing uses the internal structure of the code as the test basis. The tester has full access to the source code, logic, and paths.
📊 Statement Coverage
Every executable statement in the code is executed at least once. 100% statement coverage does NOT mean all bugs are found — just all statements ran.
🌿 Branch Coverage
Every decision point (if/else) has both TRUE and FALSE outcomes tested. Stronger than statement coverage. 100% branch → 100% statement.
🔀 Path Coverage
Every possible path through the code is tested. Most thorough but often impractical — the number of paths can be enormous in complex code.
👥 Who does it?
Primarily developers during unit testing. QA may use white-box thinking to design better black-box tests. Required for safety-critical systems.

LESSON 4.5 · COLLABORATION TECHNIQUES
Collaboration-Based Test Design — BDD and ATDD
Modern Agile teams use collaboration techniques that bring developers, QA, and business together to define tests BEFORE development begins.
🤝 ATDD
Acceptance Test-Driven Development. Team defines acceptance tests before coding. Tests become the definition of "done." Shared understanding from day one.
📝 BDD
Behaviour-Driven Development. Uses Given-When-Then format. Business-readable scenarios that serve as both documentation and automated tests. Tools: Cucumber, SpecFlow.
📋 Given-When-Then
Given a logged-in user
When they submit an empty form
Then validation error is shown
✅ Benefits
Shared understanding of requirements · Tests as living documentation · Fewer misunderstandings · QA involved from day one · Business validates logic
