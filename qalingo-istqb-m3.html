LESSON 3.1 · WHAT IS STATIC TESTING
Static Testing — No Execution Required
Static testing examines work products without executing the software. You can find defects in requirements, design documents, user stories, test plans, and source code — all before a single test is run.

This is one of the most cost-effective testing techniques. A defect found in a requirements document costs almost nothing to fix. The same defect found in production can cost 100x more.

Static testing includes two main categories: reviews (human examination) and static analysis (automated tool examination).
📄 What can be statically tested?
Requirements specs · User stories · Architecture documents · Source code · Test plans · Test cases · User manuals · Contracts
🐛 What defects does it find?
Ambiguous requirements · Missing requirements · Inconsistencies · Code standard violations · Security vulnerabilities · Unreachable code
Q — QA DETECTIVE
"The best bug is the one that never gets written. Static testing catches the defect in the blueprint before the building is constructed. Dynamic testing demolishes the building and rebuilds it to prove it was wrong. Q prefers blueprints."

LESSON 3.2 · REVIEW TYPES
The 4 Review Types — From Informal to Formal
ISTQB defines four review types, ranging from informal to highly structured. Each has different formality, roles, and objectives.
1. Informal Review
No formal process. A colleague reviews your work and gives feedback. Fast and cheap. No documentation required. Common in Agile — "can you take a look at this story?"
2. Walkthrough
The author leads the review. Reviewers ask questions and raise issues. Goal: share knowledge and find defects. Partially documented. Useful for complex features.
3. Technical Review
Peers evaluate the work product. Focuses on technical correctness. A moderator leads (not the author). Documented. Used for architecture, design, or code review.
4. Inspection
The most formal review type. Defined roles: author, moderator, reviewers, scribe. Entry/exit criteria required. Fully documented with metrics. Proven to find the most defects per hour.
Q — QA DETECTIVE
"In an Inspection, the author does NOT lead. The moderator leads. The author listens, takes notes, and does not defend their work. This is the hardest part for most developers — and the most valuable part for everyone else."

LESSON 3.3 · REVIEW ROLES
Review Roles and Responsibilities
Formal reviews have clearly defined roles. Knowing them is essential for the ISTQB exam.
👔 Manager
Decides to conduct the review. Allocates time and resources. Ensures follow-up actions are completed.
🎯 Moderator (Facilitator)
Leads the review meeting. Ensures process is followed. Does NOT have to be a domain expert. Keeps discussion on track.
✍️ Author
Created the work product. Participates in the review but does NOT lead or defend. Listens and takes notes on findings.
📝 Scribe (Recorder)
Documents all defects, issues, and decisions during the review meeting. Produces the review log.
🔍 Reviewer (Inspector)
Reviews the work product and identifies defects. Can be domain expert, QA engineer, developer, or stakeholder.
👥 Review Leader
Plans and organizes the review. Selects reviewers, distributes materials, tracks completion.

LESSON 3.4 · STATIC ANALYSIS
Static Analysis — Automated Code Examination
Static analysis uses tools to automatically examine source code or other work products without executing them. Tools can detect defects that humans easily miss at scale.

Static analysis is performed by developers as part of their development workflow — often integrated into CI/CD pipelines.
🔎 What static analysis finds
Coding standard violations · Security vulnerabilities (SQL injection, XSS) · Unreachable code · Undefined variables · Memory leaks · Overly complex code
🛠 Common tools
SonarQube · ESLint · Checkstyle · PMD · Pylint · Coverity · Fortify · SpotBugs · Semgrep
✅ Benefits
Consistent application of coding standards · Finds defects before testing · Fast — runs automatically · Scales to millions of lines of code
⚠️ Limitations
Cannot find all defects · High false positive rate possible · Cannot replace testing · Does not verify business logic correctness
Q — QA DETECTIVE
"SonarQube flagged 47 issues before the sprint demo. The developer said 'those are just warnings.' Three of them were SQL injection vulnerabilities. Static analysis warnings are evidence, Watson. Treat them as such."

LESSON 3.5 · REVIEW PROCESS
The Formal Review Process — 6 Activities
ISTQB defines a formal review process with 6 activities. These apply mainly to Technical Reviews and Inspections.
1
Planning
Define scope, objectives, roles. Check entry criteria. Distribute work products to reviewers.
2
Review Initiation
Confirm everyone has access to materials and is prepared. Check entry criteria are met.
3
Individual Review
Each reviewer examines the work product independently. Logs potential defects, questions, and suggestions.
4
Communication and Analysis
Group meeting where reviewers communicate findings. Moderator facilitates. Scribe records. Defects evaluated and logged.
5
Fixing and Reporting
Author fixes defects. Review report produced. Exit criteria checked.
6
Closing
Confirm exit criteria are met. Collect metrics. Lessons learned. Work product accepted or rejected.
